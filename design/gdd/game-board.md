# 게임 보드 (Game Board)

> **Status**: Designed
> **Author**: Luna/Solar Eclipse + Claude
> **Last Updated**: 2026-03-26
> **Implements Pillar**: 2분 안에 결판 (빠른 게임 흐름의 물리적 기반)

## Overview

게임 보드는 10열 × 20행의 2차원 정수 그리드다. 각 셀은 0(빈 칸) 또는 블록 타입 ID(1 이상)를 저장한다. 모든 게임플레이 판정(블록 충돌, 라인 클리어, 게임 오버)은 보드의 상태를 읽어 수행된다. 플레이어가 직접 상호작용하는 대상이 아니라, 테트리스 코어와 미션 조건 시스템이 읽고 쓰는 공유 데이터 구조다.

## Player Fantasy

게임 보드는 플레이어가 직접 느끼는 시스템이 아니지만, 판정의 신뢰성이 핵심이다. 블록이 어디에 고정되는지, 라인이 언제 지워지는지가 항상 명확하고 예측 가능해야 한다. 플레이어는 "내가 이렇게 놓으면 이렇게 된다"를 의심 없이 믿을 수 있어야 한다. 보드가 신뢰를 주면 플레이어는 전략에 집중할 수 있다.

## Detailed Design

### Core Rules

1. **그리드 구조**
   - 표시 영역: 10열 × 20행 (열: 0~9, 행: 0~19, 0행이 최상단)
   - 스폰 버퍼: 표시 영역 위 3행 (행: -3~-1), 플레이어에게 보이지 않음
   - 총 내부 그리드: 10열 × 23행

2. **셀 데이터 구조**
   - 각 셀은 `{ type_id: int, color: Color }` 저장
   - `type_id = 0`: 빈 칸 (color 무시)
   - `type_id >= 1`: 고정된 블록 (블록 종류 ID)
   - 낙하 중인 블록은 보드에 기록하지 않음 — 테트리스 코어가 별도 관리

3. **좌표계**
   - `(col, row)` — col은 왼쪽 0, row는 위쪽 0
   - 예: 좌하단 셀 = `(0, 19)`, 우상단 셀 = `(9, 0)`

4. **보드가 제공하는 연산**
   - `is_empty(col, row)` — 셀이 비어있는지 확인
   - `place_block(col, row, type_id, color)` — 셀에 블록 고정
   - `clear_row(row)` — 행 삭제 후 위 행들을 한 칸씩 아래로 이동
   - `is_row_full(row)` — 행이 모두 채워졌는지 확인
   - `reset()` — 보드 전체 초기화
   - `get_filled_row_count()` — 미션 진행도 추적용

### States and Transitions

| State | Entry Condition | Exit Condition | Behavior |
|-------|----------------|----------------|----------|
| **Inactive** | 게임 시작 전 / `reset()` 호출 후 | 게임 시작 시 | 모든 셀 = 0, 쓰기/읽기 불가 |
| **Active** | 스테이지 시작 | 게임 오버 또는 미션 클리어 | 블록 배치, 라인 클리어 정상 동작 |
| **Frozen** | 스킬 발동 (시간 슬로우 등) | 스킬 종료 | 보드 읽기는 가능, 새 블록 고정 불가 |
| **Game Over** | 스폰 버퍼(-3~-1행)에 블록이 고정됨 | `reset()` 호출 | 더 이상 연산 수행 안 함, 결과 화면 트리거 |

### Interactions with Other Systems

| 시스템 | 방향 | 인터페이스 |
|--------|------|-----------|
| **테트리스 코어** | 코어 → 보드 | 블록 고정 시 `place_block()` 호출. 충돌 감지 시 `is_empty()` 호출. 게임 오버 판정 시 스폰 버퍼 행 확인 |
| **테트리스 코어** | 보드 → 코어 | 라인이 꽉 찬 행 목록 반환 (`get_full_rows()`) |
| **미션 조건 시스템** | 미션 → 보드 | 진행도 추적을 위해 보드 상태 읽기 (예: 특정 색상 블록 수, 채워진 행 수) — 읽기 전용 |
| **스킬 시스템** | 스킬 → 보드 | 즉발 스킬 발동 시 보드 직접 수정 가능 (예: 폭탄 스킬 → 특정 셀 삭제). `Frozen` 상태 진입/해제 |
| **게임 HUD** | HUD → 보드 | 보드 상태를 읽어 시각적으로 렌더링 — 읽기 전용 |

## Formulas

### 라인 클리어 후 행 이동

```
클리어된 행 목록 = [r1, r2, ...] (위에서 아래 순 정렬)
각 클리어 행 r에 대해:
  r보다 위의 모든 행(0 ~ r-1)을 한 칸씩 아래로 복사
  최상단 행(0)은 빈 칸으로 초기화
```

| 변수 | 타입 | 범위 | 설명 |
|------|------|------|------|
| `cleared_rows` | Array[int] | 길이 1~4 | 이번 턴에 클리어된 행 인덱스 목록 |
| `drop_count` | int | 1~4 | 한 번에 클리어된 행 수 |

**예상 출력**: `drop_count` = 1~4 (동시에 4줄 클리어가 최대)

### 게임 오버 판정

```
is_game_over = 스폰 버퍼(-3~-1행) 중 하나라도 type_id >= 1 인 셀이 존재
```

| 변수 | 타입 | 설명 |
|------|------|------|
| `spawn_buffer_rows` | Array[int] | [-3, -2, -1] — 플레이어에게 보이지 않는 버퍼 행 |
| `is_game_over` | bool | true이면 Game Over 상태로 전환 |

## Edge Cases

| 시나리오 | 예상 동작 | 근거 |
|---------|----------|------|
| 동시에 4줄이 꽉 참 | 4줄 모두 동시에 클리어, 위 행들이 4칸 내려옴 | 표준 테트리스 규칙 (테트리스!) |
| `clear_row()` 대상이 실제로 꽉 차지 않음 | 호출 자체를 거부 (assert), 테트리스 코어가 `is_row_full()` 확인 후 호출해야 함 | 잘못된 호출은 보드 오염을 막기 위해 방어적으로 처리 |
| `place_block()` 호출 위치가 이미 채워진 셀 | 호출 자체를 거부 (assert), 테트리스 코어가 `is_empty()` 확인 후 호출해야 함 | 보드 데이터 무결성 보장 |
| 스킬로 보드 중간 행 전체 삭제 | `clear_row()`와 동일한 알고리즘으로 위 행들이 내려옴 | 일관된 물리 규칙 유지 |
| `reset()` 호출 시 Frozen 상태 중 | Frozen 해제 후 Inactive로 전환 | 상태 순서 보장 |
| 보드가 완전히 비어있을 때 `get_full_rows()` 호출 | 빈 배열 반환 | null 반환 금지 — 항상 Array 반환 |

## Dependencies

| 시스템 | 방향 | 의존 성격 |
|--------|------|---------|
| **테트리스 코어** | 테트리스 코어가 이 시스템에 의존 | Hard — 코어는 보드 없이 동작 불가 |
| **미션 조건 시스템** | 미션이 이 시스템에 의존 | Hard — 진행도 추적을 위해 보드 상태 필수 |
| **스킬 시스템** | 스킬이 이 시스템에 의존 | Hard — 즉발 스킬은 보드를 직접 수정 |
| **게임 HUD** | HUD가 이 시스템에 의존 | Hard — 보드 상태 없이 렌더링 불가 |

**이 시스템은 다른 시스템에 의존하지 않음** — Foundation Layer 확인 ✅

## Tuning Knobs

| 파라미터 | 현재 값 | 안전 범위 | 높이면 | 낮추면 |
|---------|--------|---------|--------|--------|
| `BOARD_WIDTH` | 10 | 8~12 | 블록 쌓기 여유 증가, 난이도 감소 | 공간 부족, 난이도 급증 |
| `BOARD_HEIGHT` | 20 | 16~24 | 더 많은 블록을 쌓을 수 있어 여유 증가 | 압박감 증가, 난이도 급증 |
| `SPAWN_BUFFER_ROWS` | 3 | 2~4 | 스폰 공간 여유 증가 | 게임 오버 판정이 너무 빨라짐 |

> 이 값들은 게임 전체에서 고정 상수로 사용한다. 스테이지별로 다르게 설정하지 않는다 (Anti-pillar: 과도한 랜덤 및 복잡도 배제).

## Acceptance Criteria

- [ ] 10×20 그리드가 정상적으로 초기화된다
- [ ] `place_block()` 호출 후 해당 셀의 `type_id`와 `color`가 올바르게 저장된다
- [ ] `is_row_full()` 이 꽉 찬 행에서만 `true`를 반환한다
- [ ] `clear_row()` 호출 후 해당 행이 삭제되고 위 행들이 정확히 한 칸씩 내려온다
- [ ] 4줄 동시 클리어 후 보드 상태가 올바르다
- [ ] 스폰 버퍼에 블록이 고정되면 `is_game_over`가 `true`가 된다
- [ ] `reset()` 호출 후 모든 셀이 `type_id = 0`으로 초기화된다
- [ ] `get_full_rows()`는 항상 Array를 반환한다 (비어있어도 null 반환 없음)
- [ ] 모든 상수값(`BOARD_WIDTH`, `BOARD_HEIGHT`, `SPAWN_BUFFER_ROWS`)이 코드에 하드코딩되지 않고 외부 설정에서 읽힌다
- [ ] 보드 연산이 프레임당 1ms 이내에 완료된다
