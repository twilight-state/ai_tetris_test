# 입력 시스템 (Input System)

> **Status**: Designed
> **Author**: Luna/Solar Eclipse + Claude
> **Last Updated**: 2026-03-27
> **Implements Pillar**: 2분 안에 결판 (입력 응답성이 게임 템포를 결정)

## Overview

입력 시스템은 플레이어의 키보드 입력을 감지하고, 이를 게임 액션 이벤트로 변환해 테트리스 코어에 전달하는 시스템이다. 키보드 화살표 키 및 WASD를 지원하며, DAS(Delayed Auto Shift)와 ARR(Auto Repeat Rate)로 연속 입력 타이밍을 제어한다. 입력 시스템은 어떤 게임플레이 상태인지 판단하지 않으며, 오직 "어떤 키가 눌렸는가"를 시그널로 내보내는 역할만 한다. 이 시스템 없이는 플레이어가 블록을 조작할 방법이 없다.

### 지원 액션

| 액션 | 기본 키 |
|------|---------|
| 좌 이동 | ← / A |
| 우 이동 | → / D |
| 소프트 드롭 | ↓ / S |
| 하드 드롭 | Space / W |
| 시계 방향 회전 | X / E |
| 반시계 방향 회전 | Z / Q |
| 홀드 | C / Shift |
| 일시정지 | Esc |

## Player Fantasy

입력 시스템은 플레이어가 의식하지 않아야 성공한 시스템이다. "내 손이 곧 블록이다" — 키를 누르는 순간 블록이 반응하고, 연속 입력 시 자연스럽게 미끄러지듯 이동하며, 의도한 조작이 항상 반영된다. 지연이나 씹힘 없이 원하는 대로 블록이 움직이는 것 자체가 플레이어의 집중력을 유지시키는 핵심이다. Pillar 3("2분 안에 결판")를 가능하게 하는 물리적 토대.

## Detailed Design

### Core Rules

1. **입력 처리 방식**
   - 매 프레임 Godot `Input` 싱글턴으로 키 상태 폴링
   - 키 상태: `just_pressed` (이번 프레임 눌림), `is_pressed` (누르고 있음), `just_released` (이번 프레임 뗌)
   - 버퍼링 없음 — 낙하/고정 처리 중 입력은 무시

2. **액션 → 이벤트 매핑**

| 액션 | 기본 키 | 이벤트 타입 | DAS/ARR 적용 |
|------|---------|-----------|-------------|
| 좌 이동 | ← / A | 연속형 | ✅ |
| 우 이동 | → / D | 연속형 | ✅ |
| 소프트 드롭 | ↓ / S | 연속형 | ❌ (SDF 적용) |
| 하드 드롭 | Space / W | 단발형 | ❌ |
| 시계 방향 회전 | X / E | 단발형 | ❌ |
| 반시계 방향 회전 | Z / Q | 단발형 | ❌ |
| 홀드 | C / Shift | 단발형 | ❌ |
| 일시정지 | Esc | 단발형 | ❌ |

3. **DAS/ARR 처리 (연속형 이동)**
   - 키가 처음 눌리면: 즉시 이동 1회 발생
   - DAS 타이머 시작 (기본 167ms)
   - DAS 경과 후: ARR 간격(기본 33ms)마다 이동 이벤트 발생
   - ARR = 0이면 DAS 경과 후 매 프레임 이동
   - 키를 떼면: DAS 타이머와 ARR 타이머 리셋

4. **소프트 드롭 (SDF)**
   - ↓/S를 누르는 동안 낙하 속도 × SDF 배율 적용 (기본 15x)
   - 키를 떼면 정상 낙하 속도 복귀
   - 시그널 `soft_drop_started` / `soft_drop_ended`를 발신하며, 속도 조절은 테트리스 코어가 담당

5. **출력 인터페이스 (시그널)**
   - 입력 시스템은 직접 블록을 움직이지 않음 — 오직 시그널 발신만 수행
   - 발신 시그널 목록:
     - `move_left` / `move_right`
     - `soft_drop_started` / `soft_drop_ended`
     - `hard_drop`
     - `rotate_cw` / `rotate_ccw`
     - `hold_requested`
     - `pause_toggled`

### States and Transitions

| 상태 | 진입 조건 | 탈출 조건 | 동작 |
|------|----------|----------|------|
| **Inactive** | 게임 시작 전 | 스테이지 시작 | 모든 입력 무시 |
| **Active** | 스테이지 시작 | 일시정지 / 게임 오버 / 스테이지 클리어 | 정상 입력 처리 |
| **Paused** | `pause_toggled` 발생 | `pause_toggled` 다시 발생 | Esc만 처리, 나머지 무시 |

### Interactions with Other Systems

| 시스템 | 방향 | 인터페이스 |
|--------|------|-----------|
| **테트리스 코어** | 입력 → 코어 | 시그널 수신: `move_left`, `move_right`, `rotate_cw`, `rotate_ccw`, `hard_drop`, `soft_drop_started`, `soft_drop_ended`, `hold_requested` |
| **씬 관리자** | 입력 → 씬 | 시그널 수신: `pause_toggled` |
| **씬 관리자** | 씬 → 입력 | 스테이지 시작/종료/게임오버 시 입력 시스템의 상태를 `Active`/`Inactive`로 전환 (상태 전환 트리거 소유권: 씬 관리자) |

> 입력 시스템은 어떤 시스템도 직접 호출하지 않음 — 시그널 전용 아키텍처

## Formulas

### DAS/ARR 처리 타이머

```
key_held_time += delta                          // 매 프레임 누적
if key_held_time >= DAS:
    if ARR == 0:
        emit move_event()                       // 매 프레임 즉시 이동
    else:
        repeat_timer += delta
        if repeat_timer >= ARR:
            repeat_timer = fmod(repeat_timer, ARR)
            emit move_event()
```

| 변수 | 타입 | 기본값 | 안전 범위 | 설명 |
|------|------|--------|---------|------|
| `DAS` | float | 0.167 | 0.05 ~ 0.3 | 연속 이동 시작까지의 딜레이 (초) |
| `ARR` | float | 0.033 | 0.0 ~ 0.1 | 연속 이동 간격 (초). 0 = 즉시 이동 |
| `key_held_time` | float | 0.0 | — | 현재 키를 누른 누적 시간 |
| `repeat_timer` | float | 0.0 | — | ARR 주기 내 경과 시간 |

**예상 동작**: DAS=167ms 기준으로 키를 누르고 0.167초 후 연속 이동 시작. ARR=33ms면 약 30회/초 이동 이벤트 발생.

### 소프트 드롭 속도 배율 (SDF)

```
// soft_drop_started 시그널 수신 시 — 테트리스 코어에서 처리:
effective_fall_speed = base_fall_speed * SDF

// soft_drop_ended 시그널 수신 시:
effective_fall_speed = base_fall_speed
```

| 변수 | 타입 | 기본값 | 안전 범위 | 설명 |
|------|------|--------|---------|------|
| `SDF` | float | 15.0 | 5.0 ~ 40.0 | 소프트 드롭 속도 배율 |
| `base_fall_speed` | float | 테트리스 코어에서 결정 | — | 현재 스테이지 정상 낙하 속도 |

> `effective_fall_speed` 계산은 테트리스 코어가 담당 — 입력 시스템은 시그널만 발신.

## Edge Cases

| 시나리오 | 예상 동작 | 근거 |
|---------|----------|------|
| 좌/우 키를 동시에 누름 | 이동 없음 — 양쪽 입력이 상쇄됨 | 표준 테트리스 동작. 나중에 누른 키 우선 방식은 구현 복잡도를 높임 |
| 하드 드롭 중 회전 키 입력 | 무시 — 하드 드롭 후 블록이 즉시 고정되어 조작 대상이 없음 | 버퍼링 없음 규칙 적용 |
| Paused 상태에서 이동 키 누름 | 무시 — Paused 상태는 Esc만 처리 | 상태 전환 테이블 준수 |
| ARR = 0 설정 시 DAS 전 이동 | DAS 전에는 단발 이동만 발생. DAS 경과 후 매 프레임 이동 | DAS는 연속 이동의 시작점. ARR=0은 DAS 이후에만 적용됨 |
| 소프트 드롭 중 하드 드롭 | 하드 드롭 우선 — `hard_drop` 시그널 발신, `soft_drop_ended`는 발신하지 않음 | 명시적 조작이 암묵적 조작보다 우선 |
| 게임 오버 직전 프레임에서 이동 입력 | Inactive 전환 즉시 입력 무시 — Active 상태가 아니면 처리 안 함 | 상태 전환 즉시 적용 |
| 홀드 키를 연속으로 누름 | `hold_requested` 매번 발신 — 허용 여부는 테트리스 코어가 쿨다운으로 결정 | 입력 시스템은 허용 판단 없음. 단순 시그널 발신 역할 |

## Dependencies

| 시스템 | 방향 | 의존 성격 |
|--------|------|---------|
| **테트리스 코어** | 테트리스 코어가 이 시스템에 의존 | Hard — 코어는 플레이어 입력 없이 동작 불가 |
| **씬 관리자** | 씬 관리자가 이 시스템에 의존 | Soft — 일시정지 시그널 수신 (씬 관리자가 없어도 입력 자체는 동작 가능) |

**이 시스템은 다른 시스템에 의존하지 않음** — Foundation Layer 확인 ✅

## Tuning Knobs

| 파라미터 | 현재 값 | 안전 범위 | 높이면 | 낮추면 |
|---------|--------|---------|--------|--------|
| `DAS` | 0.167s | 0.05 ~ 0.3 | 연속 이동이 늦게 시작, 의도치 않은 이동 줄어듦 | 연속 이동이 너무 빨리 시작, 의도치 않은 이동 증가 |
| `ARR` | 0.033s | 0.0 ~ 0.1 | 연속 이동 속도 느려짐, 정밀 조작 쉬워짐 | 연속 이동 속도 빨라짐. 0이면 DAS 후 즉시 벽까지 이동 |
| `SDF` | 15.0 | 5.0 ~ 40.0 | 소프트 드롭이 하드 드롭에 가까워짐 | 소프트 드롭과 정상 낙하 차이가 미미해져 유용성 감소 |

> 이 세 값은 플레이어 인게임 설정에서 조정 가능하도록 구현 권장 (접근성). MVP에서는 고정값으로 시작하고 이후 설정 UI를 추가한다.

## Acceptance Criteria

- [ ] 좌/우 이동 키를 누르면 첫 프레임에 즉시 이동 이벤트 1회 발생한다
- [ ] DAS(167ms) 경과 후 ARR(33ms) 간격으로 연속 이동 이벤트가 발생한다
- [ ] 키를 떼면 DAS/ARR 타이머가 즉시 리셋된다
- [ ] 좌/우 키를 동시에 누르면 이동 이벤트가 발생하지 않는다
- [ ] ARR = 0으로 설정 시, DAS 경과 후 매 프레임 이동 이벤트가 발생한다
- [ ] 소프트 드롭 키를 누르면 `soft_drop_started` 시그널이 발신된다
- [ ] 소프트 드롭 키를 떼면 `soft_drop_ended` 시그널이 발신된다
- [ ] 하드 드롭 키를 누르면 `hard_drop` 시그널이 1회 발신된다 (연속 발신 없음)
- [ ] 회전 키를 누르면 해당 방향의 회전 시그널이 1회 발신된다
- [ ] 홀드 키를 누르면 `hold_requested` 시그널이 발신된다
- [ ] Esc를 누르면 `pause_toggled` 시그널이 발신된다
- [ ] Paused 상태에서 이동/회전/드롭 키 입력 시 해당 시그널이 발신되지 않는다
- [ ] `DAS`, `ARR`, `SDF` 값이 코드에 하드코딩되지 않고 외부 설정에서 읽힌다
- [ ] 입력 처리 로직이 프레임당 0.1ms 이내에 완료된다
